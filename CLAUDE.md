# Groove Work — Руководство по проекту

Здесь — текущая архитектура и правила. Историю изменений не ведём (для этого есть git и `data/changelog.json`): описываем «как есть», а не «как стало».

## Принципы разработки

1. **Простое лучше сложного.** Не плодим абстракции «на будущее». Три одинаковых строки лучше преждевременной фабрики.
2. **Оптимальность и эффективность важнее тормознутости.** Никаких N+1, лишних ререндеров, избыточных re-fetch.
3. **Расширяемость и масштабируемость важнее монолита.** Новый функционал — переиспользуемые блоки (composables, components, services, repositories), не однострочные хаки.
4. **Комментарии — только там, где они реально нужны.** Не описываем «что делает» очевидный код; «почему» — только если есть скрытый инвариант, обход бага или неочевидная мотивация.
5. **Профессиональный, лёгкий и краткий код.** Понятные имена, разумная декомпозиция, без раздувания.
6. **Только архитектура цветов и токенов.** Никаких `#hex` / `rgba()` в компонентах. Только `--color-*`, `--tag-*`, `--shadow-*`, `--radius-*` (см. «Цветовая система»).
7. **Дизайн един и согласован с тем, что уже есть.** Material 3 Expressive — стиль всего приложения. Не отклоняемся от стиля действующих экранов.
8. **При новых/переписываемых разделах — сначала ищем примеры лучших UI/UX подобного функционала в интернете** (ориентир — Material 3 Expressive от Google), и реализуем по образцу, адаптируя под проект.
9. **Дублирующиеся компоненты — выносим в общий.** Если используется в двух+ местах — `components/common/`, `composables/` или `utils/`. Без копипасты.
10. **Общение между микросервисами — ТОЛЬКО через gRPC.** Контракты в `api/proto`, стабы генерит `scripts/gen_proto.sh`. Никаких REST/HTTP-вызовов сервис→сервис. (REST микросервисов — только для клиентского трафика через nginx; Redis-каналы `gw2:<svc>:events` — только доставка сокет-событий клиентам через gatewaysvc, межсервисные вызовы через них не вести.)
11. **Changelog (`data/changelog.json`) — НЕ ТРОГАТЬ без явной команды пользователя.** Когда команда дана: никаких технических деталей, технологий и стека — только пользовательские последствия («стало быстрее, надёжнее и безопаснее»), кратко, в шуточном пользовательском тоне.
12. **CLAUDE.md — не помойка.** Здесь только актуальная архитектура и правила; историю версий сюда не писать.

## Что это

Groove Work — multi-tenant платформа учёта времени задач, аналитики и коллаборации. **Идентичность развязана с компаниями:** пользователь — самостоятельная сущность (регистрируется сам, не привязан ни к чему), может состоять в нескольких компаниях с разной ролью либо ни в одной (тогда ему доступны мессенджер/звонки/профиль/контакты). Внутри компании ведутся задачи/юниты, статистика, питомцы-грувики (лёгкая геймификация) и корпоративный портал (новости/посты); мессенджер, звонки и ИИ-ассистент — кросс-компанийные по интерфейсу (данные всё равно скоупятся активной компанией). Любой пользователь создаёт свою компанию и становится её администратором. Платформой (всеми компаниями) управляет единственный супер-админ.

## Архитектура

Пятнадцать Go-микросервисов (включая шлюз, push, mail, registry, calendar, diary, notes и portal) над общей PostgreSQL (схему ВСЕХ таблиц ведёт run-once контейнер `migrate` — goose, baseline + инкременты в `back-go/migrate/migrations`) + Redis + LiveKit. Python/Flask ликвидирован полностью.

- **gatewaysvc** (`back-go/gateway`) — realtime-шлюз (наследник Flask-SocketIO): WebSocket `/ws` (кадры `{"event", "data"}`, первый кадр — `{"event":"auth","data":{"token":<PASETO>}}`, ответ `_connected`/`_error`), комнаты `all`/`user_{id}`, presence в Redis (`gw2:presence:*`: visibility + heartbeat + sweeper, `last_seen_at` → users), ринг-фаза звонков (WS-команды `call:*` → gRPC callsvc) и доставка событий ВСЕХ Redis-каналов `gw2:*:events` клиентам (общий envelope `{event, rooms, payload}`). HTTP :8096: `/ws`, exact REST `GET /api/messenger/presence`, `/healthz`. Свои события публикует в `gw2:gateway:events` — единый путь доставки, готов к нескольким инстансам.
- **callsvc** (`back-go/calls`) — вся бизнес-логика звонков + LiveKit (токены/комнаты/вебхуки) + оркестрация плашки звонка в чате (gRPC msgsvc: EnsureDialog при p2p-старте, CreateCallMessage/GetCallMessage → события `message:new`/`message:updated` через свой Redis-канал). `calls.company_id` опционален (звонки возможны и между людьми без общей компании — проставляется общая компания участников, иначе NULL). gRPC :9090 (ринг-фаза, зовёт gateway) и HTTP :8090 (REST `/api/calls/*`, вебхуки LiveKit, `/healthz`).
- **authsvc** (`back-go/auth`) — идентичность, авторизация (вкл. публичную регистрацию), компании, членство, роли и REST-бэкап. HTTP :8091: `/api/auth/*` (login/register/select-company/switch-company/refresh/logout/change-default), `/api/users/*` (платформенный список — супер-админ; directory — члены активной компании или глобальный поиск `?all=1`; me, аватарки, identicon; CRUD сотрудников активной компании — администратор компании), `/api/companies/*` (создать — любой авторизованный; список всех/toggle-active — супер-админ; карточка/настройки/члены/инвайт — администратор этой компании, проверка `companyAuthority` в сервисе; regex ai-settings — в aisvc), `/api/roles`, `/api/backup/*` (export/import ZIP — супер-админ; УНИВЕРСАЛЬНЫЙ схемо-независимый дамп: таблицы обнаруживаются из БД на лету (`to_jsonb`/`jsonb_populate_recordset`), новые таблицы попадают в бэкап автоматически; денлист транзиентных/перегенерируемых — `email_verifications`/`password_resets`/`task_embeddings`/`goose_db_version`; модалки выбора разделов на экспорт `?sections=` и восстановление (multipart-поле `sections`); ДЕСТРУКТИВНЫЙ импорт TRUNCATE RESTART IDENTITY CASCADE + FK-топосортировка + setval, всё в одной транзакции; разделы — `domain.BackupSections` ≡ `front/src/utils/backupSections.js`), `/healthz`. gRPC нет — проверка токенов у всех локальная.
- **msgsvc** (`back-go/messenger`) — мессенджер. HTTP :8092 (REST `/api/messenger/*`, кроме exact `/api/messenger/presence` — он в gateway) и gRPC :9092 (плашки звонков для callsvc: EnsureDialog/CreateCallMessage/GetCallMessage; пересылка поста портала — CreatePostMessage для portalsvc). Сокет-события → Redis `gw2:messenger:events`.
- **aisvc** (`back-go/ai`) — LLM-шлюз + деловой ИИ-ассистент. Ключи компаний (Fernet, `AI_KEY_ENCRYPTION_KEY`), вызовы ProxyAPI/OpenAI (chat + embeddings), `task_embeddings` (pgvector, семантический поиск задач) + ТВ-факт дня (фоновый goroutine-цикл раз в час по компаниям с `ai_enabled`, кэш Redis `gw2:ai:tv_fact:{cid}`). HTTP :8093 (REST `/api/companies/<id>/ai-settings*`, `/api/ai/tv-fact` и `/api/ai/assistant/{messages,history}` — см. раздел «ИИ-ассистент») и gRPC :9093 (Status/Chat/Embed/SemanticSearch/ReindexTask — зовёт tasksvc; `Chat` также используется внутрипроцессно самим ассистентом, tools-цикл — внутри aisvc, не у вызывающего).
- **tasksvc** (`back-go/tasks`) — ядро платформы: задачи (избранное, личные цвета, комментарии, ответственный/этап), юниты (1 активный на пользователя), типы юнитов, этапы, отделы, статистика (common/extended/profile/user-tasks/responsibles/employees + xlsx-экспорт на excelize) и вся YouGile-интеграция (см. раздел). HTTP :8095 (REST `/api/tasks|units|unit-types|departments|stages|stats|yougile`). Поиск задач: при включённом AI — целиком семантический через gRPC aisvc (SemanticSearch, fail-open в LIKE), реиндекс эмбеддингов — ReindexTask; хуки геймификации питомцев — gRPC petsvc (OnUnitStarted/OnUnitStopped/OnTaskClosed, fire-and-forget). Исходящий gRPC :9095 (`tasks.v1.TasksService`: GetStatsSummary/ListDepartments/GetTopEmployees/GetStatsByUnitTypes/GetStatsCalendar/SearchTasks/GetTaskLink) — тонкая обёртка над своей же статистикой/поиском для ИИ-ассистента aisvc, единственный исходящий контракт tasksvc. Сокет-события → Redis `gw2:tasks:events`.
- **petsvc** (`back-go/pets`) — питомцы-грувики, чистая гейм-механика без ИИ и без публичной ленты (см. раздел «Питомцы-грувики»). HTTP :8094 (REST `/api/pets/*`) и gRPC :9094 (хуки доменных событий tasksvc — юниты/задачи и YouGile-вебхук). Исходящих межсервисных вызовов нет. События → Redis `gw2:pets:events`; фоновый цикл заботы (проверка болезни) — внутри.
- **portalsvc** (`back-go/portal`) — корпоративный портал (см. раздел ниже). HTTP :8102 (REST `/api/portal/*`); своего gRPC-сервера нет. Исходящий gRPC — CreatePostMessage в msgsvc (пересылка поста в чат; msgsvc сам публикует `message:new` в свой канал — пуши работают). События → Redis `gw2:portal:events`.
- **pushsvc** (`back-go/push`) — пуш-уведомления (FCM, Android). HTTP :8097 (REST `/api/push/register|unregister` — токены устройств в `device_tokens`, RequireToken; `/healthz`). Подписан на Redis-каналы `gw2:messenger:events` (message:new), `gw2:tasks:events` (task:created → ответственному) и `gw2:gateway:events` (call:incoming → high-priority). Шлёт FCM HTTP v1 (OAuth2 из service-account JSON, env `FIREBASE_CREDENTIALS_JSON`; без ключа отправка отключена) ТОЛЬКО офлайн-получателям (presence `gw2:presence:online`) — приложение на переднем плане получает событие по WS (FCM-first). Мёртвые токены чистятся по ответу FCM. gRPC нет.
- **registrysvc** (`back-go/registry`) — реестры: настраиваемые компанией таблицы-справочники. У реестра есть набор полей (структура карточки 8 типов: image/file/text/number/checkbox/select/link/datetime, с раскладкой `col_span` 1..3 / `row_span` в 3-колоночной сетке) и записи. Значения записей — в `registry_records.data` (JSONB по строковому id поля) + производное `search_text` (сквозной ILIKE-поиск по тексту/числам/датам/спискам/ссылкам, триграммный GIN-индекс). HTTP :8099 (REST `/api/registries/*`): структуру (`POST/PATCH/DELETE`, `PUT …/fields`) правит администратор компании (роль 3, удаление поля чистит его данные во всех записях), записи (`…/records*`, `bulk-delete`, `uploads`, `export` — xlsx через excelize: выбранные поля, все по фильтру или только отмеченные id; картинки/файлы не выгружаются) ведёт любой участник (роль ≥1). **Внешние ссылки (read-only без авторизации):** участник создаёт/отзывает публичные ссылки (`GET/POST/DELETE /api/registries/:id/shares`, код-capability в `registry_shares`); по коду — публичные эндпоинты `GET /api/registries/shared/:code{,/records,/export}` (просмотр таблицы, карточек, экспорт; без правки), фронт-роут `/registry/:code` (`SharedRegistryView`). В Fiber middleware группы монтируется на префикс — поэтому auth-мидлварь пропускает `/api/registries/shared/*`. Скоуп — активная компания из токена; межсервисных вызовов нет (авторизация локальная, PASETO). Файлы/картинки — через `pkg/storage` (префикс `registry/`; local-том в dev, S3 в prod), отдаёт nginx `/uploads/`; удаление записи/поля чистит и сами файлы. Сокет-события (`registry:*`, `record:*`) → Redis `gw2:registry:events`.
- **calendarsvc** (`back-go/calendar`) — календари: настраиваемые компанией списки записей, привязанных к дате/времени. По структуре — близнец registrysvc (те же 8 типов полей + раскладка карточки), отличия: у каждой записи есть встроенное обязательное поле «Дата и время» без секунд (колонка `calendar_records.event_at`, по ней запись попадает в день; не входит в `calendar_fields`), а пользовательские поля поддерживают **условную видимость** (`calendar_fields.visible_field_id`/`visible_value`: поле показывается в карточке, только когда значение поля-источника совпадает с заданным — для checkbox `"true"`, для select — вариант; вычисляется на фронте, скрытые поля не сохраняются). У поля два независимых флага видимости: `show_in_table` (плитка/таблица/экспорт) и `show_in_card` (тело карточки события в видах «День»/«Неделя» и диалоге дня). Значения — в `calendar_records.data` (JSONB) + `search_text`. HTTP :8100 (REST `/api/calendars/*`): структуру (`POST/PATCH/DELETE`, `PUT …/fields`) правит администратор компании (роль 3), записи (`…/records?from&to&search` — выборка за диапазон дат для просмотра по дню/неделе/месяцу; `bulk-delete`, `uploads`, `export` — xlsx за период через excelize) ведёт любой участник (роль ≥1). Фронт-просмотр — `CalendarView` (плитки месяца/недели + хронологический список дня). **Внешние ссылки (read-only без авторизации):** `GET/POST/DELETE /api/calendars/:id/shares` (код-capability в `calendar_shares`); по коду — публичные `GET /api/calendars/shared/:code{,/records,/export}`, фронт-роут `/calendar/:code` (`SharedCalendarView`); auth-мидлварь пропускает `/api/calendars/shared/*`. Скоуп — активная компания из токена; межсервисных вызовов нет (авторизация локальная, PASETO). Файлы/картинки — через `pkg/storage` (префикс `calendar/`; local-том в dev, S3 в prod); удаление записи/поля чистит и сами файлы. Сокет-события (`calendar:*`, `entry:*`) → Redis `gw2:calendar:events`.
- **diarysvc** (`back-go/diary`) — личные ежедневники: списки записей-задач, привязанных к дню. ## Авторизация (PASETO + authsvc)

- **Access-токен** — PASETO v4.public (Ed25519, TTL 15 мин): подписывает только authsvc (`PASETO_PRIVATE_KEY`, hex seed||pub); остальные сервисы (общий `pkg/pasetoauth`) проверяют подпись по `PASETO_PUBLIC_KEY`. Скомпрометированный сервис-верификатор не может выпустить токен.
- **Refresh-токен** — PASETO v4.local (`PASETO_REFRESH_KEY`, TTL 30 дней), cookie `refresh_token` (HttpOnly, Secure, SameSite=Strict). Читает только authsvc.
- **Клеймы:** sub (id строкой), type=access, exp/iat, force_change, is_super_admin и ОПЦИОНАЛЬНЫЙ контекст активной компании (company_id, company_name, company_settings, role_level). «Нет активной компании» (company_id == nil, role_level 0) — НОРМАЛЬНОЕ состояние (мессенджер/профиль/контакты), а НЕ признак админа. Платформенный супер-админ — отдельный флаг `is_super_admin`, не роль.
- **Регистрация публичная с подтверждением email:** `POST /api/auth/register` (`{fio, email, login?, password}`) — самостоятельное создание аккаунта без компании. Логин предлагается транслитом из ФИО (`GET /api/auth/suggest-login?fio=…`: 6 букв фамилии + `.` + буква имени + буква отчества, фронт подставляет в редактируемое поле; пустой логин сервис генерит сам, коллизия → числовой суффикс), пароль фронт генерит на клиенте. Регистрация сессию НЕ выдаёт: создаёт `users.email_verified=FALSE`, пишет код+токен в `email_verifications` и шлёт письмо через mailsvc, ответ `{status:"verification_required", email}`. Подтверждение — `POST /api/auth/verify-email` (`{token}` из ссылки `/{origin}/verify-email?token=…` ИЛИ `{email, code}`): ставит `email_verified=TRUE`, удаляет запись, выдаёт сессию (как login). Переотправка — `POST /api/auth/resend-verification` (троттл 60 с, тихо). Логин неподтверждённого аккаунта → 403 `EMAIL_NOT_VERIFIED` (фронт ведёт на экран кода). Существующие аккаунты и сотрудники, заведённые администратором (force-change вместо верификации), считаются подтверждёнными (`email_verified=TRUE`). Login при 0 компаний не падает (сессия без активной компании); 1 компания — автоактивна; >1 — login-gate (select-company).
- **Сброс пароля по email:** `POST /api/auth/forgot-password {email}` (ответ всегда ok — не раскрываем аккаунт; троттл 60 с) → токен в `password_resets` + письмо `reset_password` через mailsvc со ссылкой `/{origin}/reset-password?token=…`. `POST /api/auth/reset-password {token, new_password}` → меняет пароль, гасит токен, возвращает `{login}` (фронт ведёт на login без автологина, экраны `/forgot-password`, `/reset-password`).
- **Раздел «Компании» (управление):** маршруты фронта `/companies` (список: супер-админ — все компании платформы; обычный — `GET /api/companies/mine`, где он администратор/создатель) и `/companies/:id` (`CompanyManageView` — табы Обзор/Участники/Настройки/Опасная зона). Настройки компании (ИИ, выходные, режим питомцев-грувиков, ссылка-приглашение, фичи, YouGile) переехали сюда из `SettingsView` (там остались только платформенно-личные: тема, личный YouGile, справка, о приложении, резервная копия для супер-админа). **Управление участниками/ролями и создание-редактирование сотрудников компании (`POST/PATCH /api/companies/:id/users`, `…/members*`, invite-перевыпуск, удаление компании) — только СОЗДАТЕЛЬ компании (`companies.created_by`) или супер-админ** (`creatorAuthority` в сервисе); не-создатель-администратор видит компанию и правит её настройки, но участниками не управляет (`companyAuthority` — для чтения/настроек). Sidebar показывает пункт «Компании» при `canManageCompanies` (админ ≥1 компании или супер-админ). **Приглашения в компанию — двумя путями:** ссылка-код `/join/:code` (любой → роль Сотрудник) и **email-инвайт с ролью** (`POST /api/companies/:id/invites {email, role_id}` — создатель/супер-админ; токен в `company_invites`, письмо `company_invite` через mailsvc со ссылкой `/{origin}/invite/:token`; получатель: `GET /api/companies/invites/:token` превью + `POST …/accept` → членство с ролью + сессия; токен — capability, как ссылка-код).
- **Фронт токен НЕ декодирует** — authsvc дублирует клеймы в телах ответов register/login/refresh/change-default; стор кладёт их в `claims` (`applySession` в `stores/auth.js`); client.js обновляет их при каждом refresh. 401 → очередь запросов + refresh + повтор.
- **force_change:** пользователь с `is_default_pass=TRUE` получает `force_change: true` — все API возвращают 403 `FORCE_PASSWORD_CHANGE`, кроме `/api/auth/change-default` и logout. Дефолтный пароль — `<login>123`.
- **Брутфорс-щит** — в authsvc (Redis `gw2:bf:attempts:{login}` / `gw2:bf:locked_until:{login}`): после каждых 5 неудач блокировка 10·2^(n−1) секунд, ответ 429 `{retry_after_sec}`, фронт показывает таймер на LoginView. Redis недоступен → fail-open.
- **Пароли** — pgcrypto в PostgreSQL: `crypt(pw, gen_salt('bf'))`, проверка `crypt(pw, hash) = hash`.
- **Аватарки** — общий uploads-volume (`UPLOAD_FOLDER/avatars/`), наружу отдаёт nginx `/uploads/` (в dev — vite-плагин serve-uploads); `avatar_path = NULL` → identicon `GET /api/users/<id>/identicon` (PNG 8×8 pixel-art от sha256(id), генерит authsvc).
- **Dev-ключи PASETO** захардкожены синхронно в: `dev.sh`, `Makefile`, `deploy/docker-compose.override.yml` (public `15ef4397…3fe1`). Прод-ключи генерирует `deploy_server.sh` (пара Ed25519 — целиком, чтобы публичный соответствовал приватному).
- Все api-модули `front/src/api/*.js` ведутся вручную (Go-сервисы Swagger не публикуют; генератор gen-api.mjs удалён вместе с Flask).

## Система прав (идентичность развязана с компаниями)

**Пользователь — самостоятельная сущность** (таблица `users`: только идентичность — login/ФИО/пароль/контакты/аватар, `is_active`, `is_super_admin`). Он НЕ знает про компании. Принадлежность и роль живут ТОЛЬКО в связке `user_companies` `(user_id, company_id, role_id, post)`. Один аккаунт — во многих компаниях с разной ролью; «нет компании» — валидное состояние. Любой пользователь может создать компанию (`POST /api/companies`) и стать её администратором.

**Три роли В КОМПАНИИ** (фиксированы в БД, создавать/удалять нельзя; `domain.Level*`, фронт — `composables/usePermission.js`, уровень из `auth.roleLevel` = роль в активной компании):

| Уровень | Роль | Особенности |
|---|---|---|
| 1 | Сотрудник (EMPLOYEE) | базовая работа с задачами/юнитами |
| 2 | Менеджер (MANAGER) | +управление чужими юнитами, отделы/типы юнитов, экспорт статистики |
| 3 | Администратор (ADMIN) | +управление компанией: настройки, члены (add/role/remove), инвайт, CRUD сотрудников |

**Супер-админ — отдельный класс** (`users.is_super_admin`, единственный, бутстрап — `reset_superadmin_password.sh`/миграция из бывшего `is_root_admin`): видит все компании и всех пользователей, управляет компаниями (`GET /api/companies`, toggle-active, delete), но к компанийному контенту (задачи, грувики, YouGile, статистика) доступа НЕ имеет. Не является ролью и не проходит `RequireRole`.

**Активная компания** сессии (выбранная при login/switch) — в access-токене (`company_id`, `role_level`); переключение — `switch-company`. Хака `?company_id=` для админа БОЛЬШЕ НЕТ.

**Middleware (`pkg/pasetoauth`):** `RequireAuth` (любой залогиненный) → `RequireRole(level)` (нужна активная компания с ролью ≥ level — супер-админ не проходит) / `RequireSuperAdmin` (платформа). Отключённая компания → 403 `COMPANY_DISABLED`.

**Гарды управления** (в `back-go/auth/internal/service`): роль не выше своей (равную — можно); супер-админа изменять нельзя; защита «последнего администратора компании»; управление членами компании — её администратор (роль 3) или супер-админ (`companyAuthority`). Создатель компании автоматически получает членство-администратора.

## Ключевые бизнес-правила

- У пользователя единовременно только 1 активный юнит (`datetime_end IS NULL`); нельзя архивировать задачу с активным юнитом.
- Удаление типа юнита каскадно удаляет все юниты с этим типом.
- Цвет задачи индивидуален для пользователя (`user_task_colors`); в сокет-броадкастах `task:created/updated` поле `color` вырезается, чтобы чужие клиенты не перезаписали свой (в tasksvc — `dto.TaskBroadcast`).
- Собственные действия пользователя на фронте — оптимистичные обновления стора; сокет-события дублируют их для остальных (handlers идемпотентны).

## WebSocket и presence

Realtime — gatewaysvc (`back-go/gateway`), фронт подключается тонкой WS-обёрткой `front/src/socket/gateway.js` (socket.io-подобный API: on/emit/connected/disconnect; реконнект 1с→5с; emit при обрыве буферизуется и уходит после повторной авторизации). Протокол — JSON-кадры `{"event", "data"}`: первый кадр клиента `{"event":"auth","data":{"token":<PASETO>}}` (10с таймаут), сервер отвечает `_connected` (диспатчится слушателям как `connect`) или `_error` + close; дальше события в обе стороны как есть; пинг/понг WS — 25с/60с. Сервер присоединяет к комнатам `all` и `user_{id}`. Мутации приходят из сервисов через Redis-каналы `gw2:<svc>:events` (envelope `{event, rooms, payload}`) и доставляются вербатим; company-scoped события несут `company_id` — клиент фильтрует в сторе. Presence — в Redis (`gw2:presence:beats` ZSET + `gw2:presence:online` SET, мульти-инстанс-ready): онлайн = есть живая видимая вкладка (`presence:visibility` + heartbeat каждые 25с + sweeper раз в 15с, простой >60с — офлайн), `last_seen_at` пишется в users на переходе в офлайн, `presence:update` — только на переходах.

## Звонки (callsvc + LiveKit)

- Весь медиа-транспорт — LiveKit (`livekit/livekit-server:v1.9`, сигнальный WS через nginx `/livekit`, медиа-порты 7881/tcp + 7882/udp). Бизнес-логика — callsvc. NAT/VPN/мобильные сети, где прямой UDP закрыт, пробивает **встроенный в LiveKit TURN-relay** (только прод-оверлей: `turn.enabled`, TLS 5349 + UDP 3478, cert — общий Let's Encrypt-том nginx, домен `LIVEKIT_TURN_DOMAIN`); LiveKit сам анонсирует TURN клиентам с эфемерными кредами — правок в callsvc/фронте/Android не требуется, отдельный диапазон relay-портов не нужен (relay терминируется в SFU). Внешний coturn не используем.
- **gRPC-контракт** `calls.v1.CallService` (`back-go/calls/api/proto/calls/v1/calls.proto`): StartCall / InviteToCall / AcceptCall / DeclineCall / LeaveCall / EndCall. Транспорт всегда OK; бизнес-ошибка — поле `error {code, message, http_status}`. Ответы несут готовый снапшот `Call` и списки адресатов — gateway эмитит сокет-события не читая БД. Стабы: `scripts/gen_proto.sh` (Go → `back-go/pkg/gen/<svc>pb`), результат коммитится.
- **gateway — тонкий шлюз ринг-фазы:** WS-команды `call:*` → `internal/ring` → gRPC callsvc (`CALLS_GRPC_ADDR`); недоступность → `call:error {code:'CALLS_UNAVAILABLE'}`. Оркестрация плашки звонка — в callsvc: при p2p-старте сам создаёт парный диалог (gRPC msgsvc EnsureDialog), плашку `kind='call'` (CreateCallMessage → `message:new`) и обновляет её на каждом изменении статуса (GetCallMessage → `message:updated`) — fire-and-forget горутины, плашка звонок не роняет.
- **Обратный канал:** callsvc публикует `call:ended` / `message:updated` в Redis `gw2:calls:events` в общем envelope — gateway доставляет клиентам; так события вебхуков LiveKit доезжают до клиентов.
- Ринг-state — in-memory в callsvc (`internal/ringstate`), восстанавливается из БД+LiveKit (`ReconcileStartup` и лениво в вебхуках). Лимит 9 участников, гости считаются в нём же. Identity: `u{user_id}` (metadata `{user_id, avatar_path}`), гости `g-{hex}` (`{guest:true}`).
- **Ссылки-приглашения:** `/{origin}/call/<share_code>` — публичный роут; `GET/POST /api/calls/join/<code>` (гость представляется именем, авторизованный входит под собой — optional JWT-заголовок).
- **Фронт:** `services/livekit.js` (`CallRoomManager` поверх livekit-client), `stores/call.js` (фазы idle/incoming/outgoing/active, чат звонка — data-канал topic `chat`, outgoing-таймаут 45с, guard `handleEnded` по `call_id`), `CallView.vue` (mini-режим, перетаскивание, демонстрация экрана, панели участников/чата), `CallAudioSink.vue` — звук всех удалённых.
- Сокет-события клиенту: `call:started {call, livekit:{token,url}}`, `call:incoming`, `call:accepted`, `call:invited`, `call:ended` и т.д.

## Мессенджер

Весь домен — в msgsvc (`back-go/messenger`). Диалоги 1:1 (`conversations`, уникальная пара user_a<user_b; `company_id` НУЛЛАБЛЕ — переписка между людьми без общей компании разрешена) + dev-чат поддержки (адресаты — супер-админы; pet-чата больше нет — питомцы полностью отвязаны от ИИ и от мессенджера, см. «Питомцы-грувики»). Сообщения: текст (Markdown-подсветка по выделению), вложения до 25 МБ (общий uploads-volume), ответы (`reply_to_id`, SET NULL), пересылка (файлы копируются физически), закрепление сообщений (общее) и чатов (личное), soft-delete «у себя/у всех» (обе стороны скрыли → физическое удаление + чистка файлов), прикрепление задач (`kind='task'` + `TaskPreview`) и постов портала (`kind='post'` + замороженный `PostPreview{id,title,excerpt,cover_url}` — не живой JOIN), запись экрана. REST `/api/messenger/*` (msgsvc :8092); сокет-события публикует в Redis `gw2:messenger:events` — gateway доставляет их в WS-комнаты вербатим. gRPC msgsvc: EnsureDialog/CreateCallMessage/GetCallMessage (callsvc, плашки звонков), CreatePostMessage (portalsvc, пересылка поста — создаёт сообщение `kind='post'` в существующем/новом диалоге).

Важные инварианты:
- Прочтение: открытый+сфокусированный чат отмечает read сразу; плюс на возврат фокуса вкладки и при отправке. `activeConversationId` общий у MessengerView и MiniMessenger.
- **Плавающий хаб** (`MiniMessenger.vue`) — глобальный FAB поверх всего (z-index 10050, выше ActiveUnitModal), ДВЕ вкладки: «Ассистент» (см. «ИИ-ассистент», дефолтная) и «Сообщения» (весь функционал мессенджера как есть). Видим ВЕЗДЕ, включая мобильные и `/messenger` — скрывается только на fullscreen-роутах (`/tv`, активный полноэкранный звонок). Независимый слой от плавающего питомца (`FloatingPet.vue`, левый нижний угол, z-index ~10055) — оба монтируются в `App.vue`.
- Поле ввода (`MessageInput.vue`): на десктопе Enter отправляет, Shift+Enter — перенос; на тач-устройствах (`(hover: none) and (pointer: coarse)`) Enter — перенос строки, отправка только кнопкой.
- Уведомления: Web Notifications через Service Worker (`public/sw.js`) + Web Audio «бип»; разрешение и разогрев AudioContext — по первому жесту.

## Питомцы-грувики

Чистая гейм-механика без ИИ, без публичной ленты, без соцсети — весь домен в petsvc (`back-go/pets`), все таблицы company-scoped, гейт компании — `GrooveEnabled`/`uses_groove` (имя настройки в authsvc не переименовано, это техдолг; пользователю показывается как «Питомцы»). **Плавающий компаньон, а не отдельная страница-лента:** `FloatingPet.vue` монтируется глобально в `App.vue` (draggable, снап к ближайшему краю экрана, позиция в `localStorage`, z-index ~10055 — выше `ActiveUnitModal`; speech bubble только на переходах состояния — проголодался/заболел, не по таймеру), тап открывает `PetDetailModal.vue` (градиентный хэдер по образцу карточек сотрудников, действия, приватная история). Полноценная страница `/pets` (`PetsView.vue`) — хаб для магазина/питомцев коллег/рейтинга, которым тесно в мини-модалке.

**Экономика:** XP (эволюция по порогам `domain.StageXP`; `applyEvolution` пересчитывает характер и ПРИРОДНЫЙ вид — купленный в магазине облик эволюция не сбрасывает, новый природный вид лишь разблокируется) и валюта **кудосы** (переименованы из «грувов», колонка `pets.kudos`) — начисляются НАПРЯМУЮ за работу, с дневными капами по источнику в Redis (fail-open, `AwardKudos`/`AwardXP`): юнит (`OnUnitStopped` — 1 кудос/5 мин кап 15, 1 XP/3 мин кап 40), закрытая задача (`OnTaskClosed` — 5 кудосов кап 25, 8 XP кап 40). Хуки от tasksvc — единственный входящий gRPC petsvc (fire-and-forget после коммита, ошибки только в лог). **Read-эндпоинты не пишут в БД впустую** (`GetMyPet`/`BumpQuest` сохраняют, только если что-то реально изменилось — квест-ролловер или впервые определённый характер) — иначе частый поллинг клиента гоняет read-modify-write и рискует lost-update перезаписать конкурентное начисление хука устаревшим снимком (см. `TestPetsUnitStoppedAwardsKudosAndXP` в apitest).

**Кормление/прогулка/лечение** — платные (кудосы) действия с мини-играми на фронте (`components/pets/{Feed,Walk,Heal}MiniGame.vue`): кормление (drag еды в рот, `FeedPet`, сытость даёт XP-буст ×1.5 до конца дня), прогулка (тайминг-мини-игра, `WalkPet`, лечит понемногу если питомец болен), лечение (серия точных тапов, `HealPet`, только если болен — дополняет БЕСПЛАТНОЕ пассивное лечение работой, не заменяет). **Поглаживание чужого питомца** — тап по карточке коллеги на `/pets` открывает мини-игру «трение ладошкой» (`StrokeMiniGame.vue`: водишь указателем по питомцу, `createRubTracker` копит дистанцию, каждый полный цикл — один платный `StrokePet` с сердечками; при исчерпании дневного лимита — финал «наглажен до завтра» и автозакрытие). Бэкенд: `StrokePet` — таблица `pet_strokes`, лимит 3/день на пару гладящий+питомец (`STROKED_ENOUGH`), 1 кудос у гладящего, XP/настроение получателю без кудосов. `/pets` (`PetsView.vue`) — единая страница без вкладок: мой питомец+квест, питомцы коллег, рейтинг одним скроллом; «Магазин» — диалогом (query `?tab=` совместим: shop — открыть магазин, colleagues/rating — прокрутка к секции). **Удаление питомца сотрудника** — администратор компании (роль 3, проверка уровня в сервисе): `DELETE /api/pets/zoo/:user_id` (company-scoped; транзакцией чистятся `pet_strokes`/`pet_shop_purchases`/`pet_kudos_weekly`, история — каскадом FK; событие `pet:deleted` в комнату all), кнопка с подтверждением на карточке коллеги; свой питомец у владельца пересоздаётся штатно первым `GetMyPet`. Болезнь — простой в РАБОЧИХ днях компании (`settings.weekend_days`, выходной не считается и не заболевает).

**Магазин** (`pet_shop_items`/`pet_shop_purchases`, БД вместо Go-констант — ассортимент правится данными, без деплоя) — 4 механики нестатичности одновременно: постоянные + ротационные (`active_from/active_to`) товары, редкость (`common/rare/epic/legendary`, цветная рамка карточки), ограниченный тираж на компанию (`limited_quota`, остаток считается по `pet_shop_purchases`), достижения (`unlock_kind=achievement`, не продаются за кудосы) — плюс бесплатный ежедневный «сюрприз» (`GET /shop/mystery`, взвешенный по редкости, БЕЗ оплаты — не лутбокс).

**Развитие после максимальной формы** (миграция 00033): **престиж** — «Легенда» может переродиться (`POST /api/pets/pet/prestige`): `pets.generation`+1, стадия/XP в ноль, вид снова яйцо; кудосы/гардероб/купленные виды/домик сохраняются; атомарный узкий UPDATE с гейтами в WHERE (макс. стадия, здоров, не в пути); поколения ≥2 разблокируют эксклюзивные виды `domain.PrestigeSpecies` (phoenix/alien/robot — не продаются); поколение видно в бейджах (модалка/зоопарк/рейтинг). **Сезонный трек** — кудосы, заработанные за календарный квартал МСК (`pet_kudos_seasonal`, ключ «2026-Q3», инкремент рядом с недельным счётчиком), открывают пороги `domain.SeasonTrack` (награды: аксессуар/декор/кудосы); `GET /season`, `POST /season/claim {threshold}` — «награда один раз» гарантирует PK `pet_season_claims`; новый квартал — трек с нуля; фронт — `SeasonTrackCard` на `/pets`. **Домик** — кудос-сток: каталог декора `domain.HouseDecor` (Go-константы; цена 0 = только с трека), покупка `POST /house/buy` (атомарный UPDATE: списание+jsonb-append под guard'ами), СВОБОДНАЯ расстановка drag'n'drop (`pets.house_placed` jsonb `[{key,x,y}]` в % сцены, `POST /house/arrange`, лимит `HousePlacedMax`; `HouseItem.UnmarshalJSON` принимает и легаси-строку); домики видны коллегам (кнопка 🏠 на карточке зоопарка → `PetHouseDialog` в режиме `guestPet`). Поля престижа/домика в full-row `SavePet` намеренно не входят — только узкие атомарные методы.

**Приключения** (appointment-механика Finch): `POST /api/pets/pet/adventure` — бесплатно отправить здорового питомца в случайную локацию на 2–4 часа (кап 3 старта/день, Redis-источник `adventure`); в пути платные действия и поглаживания дают 422 `PET_AWAY`, на виджете бейдж 🧭; возврат фиксируется ЛЕНИВО первым `GetMyPet` владельца после срока (атомарный UPDATE c RETURNING — награда ровно один раз): вариативно 3–10 кудосов + 5–15 XP через `AdjustBalances`, разовое поле ответа `adventure_reward`. **Рейтинг** — кудосы, заработанные с начала текущей ISO-недели (`pet_kudos_weekly`, инкрементируется в `AwardKudos`; сортировка рейтинга — по ним же), `GET /api/pets/rating` — топ-10 + позиция зрителя (`RatingCard.vue`). **Приватная история** (`pet_activity_log`) — узкая замена публичной ленты: видна ТОЛЬКО владельцу во вкладке «История» модалки, никуда не публикуется. **Убрано полностью** (вместе с бэкендом): публичная лента `feed_events`/кудосы-как-реакция/рейд недели/wrapped/AI-чат-бот питомца (pet-чат в мессенджере убран, `is_pet_chat` не существует)/погода Грувика (была нужна только как повод для чат-реплик). REST `/api/pets/*` (petsvc :8094), сокет-события в `gw2:pets:events` (личные — комната `user_{id}`).

## Корпоративный портал

Независимый микросервис portalsvc (`back-go/portal`, HTTP :8102, REST `/api/portal/*`), company-scoped. **Фронт — единый раздел-хаб**: один пункт «Портал» в навигации, внутри вкладки «Лента» (`/portal`, `PortalView`) и «Сотрудники» (`/employees`, `EmployeesView`) — общий переключатель `PortalHubTabs.vue`, пункт активен на обоих роутах. **Никак не связан с питомцами** — только контент и коллаборация, по образцу лучших практик интранетов (Viva Engage/SharePoint/LinkedIn): посты (`portal_posts`, опциональный заголовок, тело, привязка к разделу), тематические разделы (`portal_topics` — CRUD только администратор компании, роль 3), плоские комментарии (`portal_comments`, без вложенности — стандарт для новостной ленты, не обсуждения), реакции (`portal_reactions`, набор эмодзи), вложения (`portal_attachments`, через `pkg/storage`, префикс `portal/`; удаление — `DELETE /api/portal/attachments/:id`, автор поста или админ, чистит файл). **Тело постов и комментариев — Markdown**: рендер общий `components/common/MarkdownView.vue` поверх `utils/markdown.js` (безопасный парсер без либ: заголовки h1-h3, списки/чек-листы, цитаты, таблицы, код/фенсы, картинки, автолинки; XSS невозможен — HTML экранируется); в превью пересылки (`postPreview` в forward.go) разметка вычищается `stripMarkdown`. Длинный пост в ленте сворачивается по фактической высоте («Показать полностью»); картинки — натуральных пропорций, по клику `ImageLightbox` (общий, с галереей-стрелками). Композер (`PostComposer`) — в стиле соцсетей: шапка автор+раздел, безрамочные поля, MD-тулбар с предпросмотром и ПКМ-меню форматирования на выделении, сетка превью медиа с удалением и canvas-редактором картинок (`components/common/ImageEditDialog.vue` — обрезка/поворот/отражение, в т.ч. повторно для уже загруженных), prop `preset {title, body}` — публикация заметки/выделения из раздела «Заметки». Закрепление (`pinned_at`/`pinned_by` + опциональный срок `pinned_until`: `POST /posts/:id/pin {days: 1..30|null}`, истёкший пин трактуется фильтром как незакреплённый) — автор поста или администратор, **лимит 10 актуально закреплённых на компанию** (атомарно под advisory-локом). **Лента — keyset-пагинация**: `GET /posts?limit&cursor` → `{pinned, posts, next_cursor}` (pinned — все актуальные, только на первой странице; cursor — opaque base64url от (created_at,id)); фронт грузит дальше кнопкой «Показать ещё» (`fetchMore`). **Непрочитанные**: `portal_seen(user_id, company_id, seen_at)`, `GET /unread` + `POST /seen`; бейдж на пункте «Портал» в сайдбаре/нижней навигации, сокет `post:new` наращивает вживую, заход в ленту сбрасывает. Поиск — триграммный GIN-индекс по заголовку+телу (`pg_trgm`, как у registry/calendar). **Пересылка поста в мессенджер** — `POST /posts/:id/forward {conversation_ids?, user_ids?}` → gRPC `CreatePostMessage` в msgsvc (по образцу `CreateCallMessage` у callsvc): создаёт сообщение `kind='post'` с замороженным превью (`title/excerpt/cover_url`) в существующем/новом (`EnsureDialog`) диалоге — рендерится как «post-pill» в `MessageBubble.vue`, клик ведёт на `/portal/:id`. Сокет-события (`post:*`, `comment:*`, `reaction:*`, `topic:*`) → Redis `gw2:portal:events`.

## YouGile-интеграция

Целиком внутри tasksvc (`back-go/tasks`): инфраструктура — `internal/yougile` (HTTP-клиент REST v2 с ретраями 429/5xx, парсер ссылок на карточки, Fernet-крипто), бизнес-логика — `internal/service/yougile_*.go`, REST `/api/yougile/*` (формы ответов байт-в-байт с прежним Flask-блюпринтом, ошибки валидации — `{"error": "VALIDATION", "details": …}`). Per-company настройка (администратор компании подключает компанию/проект/доску; личные ключи пользователей шифруются Fernet — `YOUGILE_ENC_KEY` в env tasksvc, ротация ключа = потеря привязок). Импорт карточек по короткой ссылке (резолв `OIP1-2454` перебором колонок + BFS по подзадачам), экспорт задач, отвязка; системные комментарии в обе стороны. Двусторонняя синхра: исходящий пуш — внутренний вызов tasksvc после update/archive/restore (горутина, best-effort); входящая — публичный вебхук `POST /api/yougile/webhook/<companyId>/<secret>` (нужен `YOUGILE_WEBHOOK_PUBLIC_BASE` в env tasksvc; constant-time проверка секрета, неверный — 404). Антицикл: `yougile_sync_hash` (sha1 от title|deadline_ms|completed) — вебхук игнорит собственное эхо. Инвариант: задача с активным юнитом не архивируется даже по completed из YG. Закрытие из вебхука зовёт petsvc OnTaskClosed.

## Цветовая система фронтенда

`front/src/assets/tokens.css` — Material You Expressive / M3, слои:
1. `--ref-*-h/c/l` — параметры цвета (пишет `theme.js`)
2. `--_p-*`, `--_s-*`, `--_n-*` — тональные палитры OKLCH (нейтральная гамма — от необязательного `neutral` темы)
3. `--color-*` — семантические токены (primary, surface, error, success…)

`[data-dark="true"]` — тёмная тема. Режим оформления: light | dark | **system** (следует за `prefers-color-scheme`, живое переключение) — `stores/theme.js`, localStorage `gw_theme_mode`. Для старых iOS без oklch — hex-фолбэк через `@supports not (color: oklch(0 0 0))` в конце tokens.css.

**Градиент фона приложения** (концепция «Expressive Glass», `front/DESIGN.md`): фон `.app-layout` — 4 радиальных пятна из цветов темы, включён по умолчанию; тумблер и генерация вариантов — в настройках внешнего вида. theme.js хранит рецепт (роль токена/позиция/размер/доля, localStorage `gw_bg_gradient`) и пишет слои в `--bgg-1..4`; цвета — `color-mix` от `var(--color-*)`, т.е. градиент сам следует за темой и тёмным режимом (`--bg-grad-dim`). Всё стекло (`--acrylic-*`) держится на этом просвечивающем градиенте: `--acrylic-card-bg` = плотность `--acrylic-bg`, но без blur (blur — плавающим слоям и крупным панелям-каркасам). Экраны входа/регистрации — общий сплит-каркас `components/auth/AuthShell.vue`.

**Цвета-теги задач:** 8 пастельных цветов, токены `--tag-<name>-surface/-border/-accent`; набор продублирован в `front/src/utils/taskColors.js` и `domain.TaskColors` tasksvc.

**Правило:** никаких `#hex`/`rgba()` в компонентах — только токены.

## Версионирование

**Единый источник истины на сервере, без хардкода в клиентах:**
- **Версия продукта** = первая запись `data/changelog.json` (`versions[0].version`). И фронт, и мобильное приложение показывают её, подтягивая с сервера (фронт — через `/api/changelog`, composable `useChangelog().latestVersion`). В бандл фронта версия НЕ зашивается (импорт `package.json` убран); `front/package.json` хранит её лишь как npm-метаданные. `versionName` обоих Android-проектов Gradle читает из `data/changelog.json` при сборке (regex первой `version`).
- **Мобильное приложение (Capacitor, `mobile/`)** = `apps/mobile/version.json` (`current_build`, формат ГГММДДН — versionCode; читает Gradle при `make apk`). Канал раздачи `/apps/mobile/` (зеркало `/apps/desktop/`): сюда смотрят карточка «Скачать APK» на сайте, всплывашка устаревшего нативного приложения и автопроверка обновлений самой обёртки (`UpdateChecker`, раз в 6 часов, диалог со скачиванием). При релизе APK: поднять `current_build` → `make apk` → `make deploy-apk`. Подпись/applicationId неизменны со времён нативного приложения — APK встаёт обновлением поверх.
- **Прежний нативный канал** = `apps/version.json` + `/apps/groovework.apk` — ЗАМОРОЖЕН на переходной nag-сборке (`make apk-legacy` из `GrooveWorkAndroid/`): старые установки обновляются до неё штатной тихой проверкой, она показывает обязательную модалку «приложение устарело» (повтор каждые 10 минут), которая скачивает и ставит APK нового канала. После миграции пользователей канал не трогать.
- **Десктоп-клиент (Electron)** = `apps/desktop/version.json` (`current_version` semver из `desktop/package.json` + имена артефактов). Пишется `make desktop` (сборка dmg/exe/AppImage → `apps/desktop/`), заливается `make deploy-desktop`. Обёртка сверяет свою версию с этим файлом при старте и раз в 6 часов и предлагает скачать установщик; сам UI приезжает с прод-сервера при обычном деплое. Карточка скачивания — «О приложении» (скрыта внутри Electron и на мобильных).

Мини-версии за фиксы одного релиза не плодим. Правила changelog — принцип 11.
